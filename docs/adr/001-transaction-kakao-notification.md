# ADR-001: 카카오 알림을 트랜잭션 내부에서 호출하되 best-effort로 유지

## 상태
승인됨

## 맥락
`OrderService.createOrder()`에 `@Transactional`을 추가하면서, 카카오 알림 전송(`KakaoNotificationService.sendOrderNotification`)의 위치를 결정해야 했다.

### 선택지

| # | 방안 | 장점 | 단점 |
|---|------|------|------|
| 1 | 트랜잭션 내부에서 호출 (현재) | 코드 단순, 주문 흐름이 하나의 메서드에 완결 | 트랜잭션 롤백 시 이미 전송된 알림 회수 불가 |
| 2 | 트랜잭션 커밋 후 호출 (`@TransactionalEventListener`) | 롤백 시 알림 미전송 보장 | 이벤트 리스너 도입 필요, 복잡도 증가 |
| 3 | 별도 비동기 처리 (`@Async`) | 트랜잭션과 완전 분리 | 비동기 인프라 설정 필요, 에러 추적 어려움 |

## 결정
**방안 1**: 트랜잭션 내부에서 호출하되, `KakaoNotificationService`가 예외를 전파하지 않도록 try-catch + 경고 로그로 감싼다.

## 근거
- 카카오 알림은 주문의 핵심 비즈니스 로직이 아닌 부가 기능이다.
- `KakaoNotificationService`는 이미 내부에서 예외를 잡아 로그만 남기므로, 알림 실패가 트랜잭션 롤백을 유발하지 않는다.
- 롤백이 발생하는 시점은 알림 전송 이전 단계(재고 차감, 포인트 차감)이므로, 실제로 "주문 실패인데 알림이 전송되는" 상황은 발생하지 않는다.
  - 재고 차감 실패 → 알림 도달 전 롤백
  - 포인트 차감 실패 → 알림 도달 전 롤백
  - 주문 저장 성공 후 → 알림 전송 (이 시점에서 롤백 원인이 없음)
- 이벤트 리스너나 비동기 처리는 현재 규모 대비 과도한 복잡도를 추가한다.

## 결과
- 트랜잭션 경계는 `createOrder()` 전체를 감싸되, 카카오 알림 실패가 주문을 방해하지 않음
- 향후 알림 정합성이 중요해지면 방안 2(`@TransactionalEventListener`)로 전환 검토
